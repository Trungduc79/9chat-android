1. Tóm tắt clean code

Module kho được xây theo các nguyên tắc sau — giữ nguyên khi port:

  • Surgical / single-responsibility: mỗi endpoint làm 1 việc rõ ràng (confirm duyệt, fulfill giao, show đọc). Không gộp business logic vào controller list.
  • Snapshot pattern: mỗi order_item lưu snapshot (tên SP, phân loại, đơn vị, conversion_factor, giá) → UI không phụ thuộc catalog hiện tại; tên/đơn vị luôn đúng tại thời điểm tạo đơn.
  • Tồn kho nhất quán giao dịch: khi đổi SL giao, BE revertOneItem() (hoàn tồn cũ) rồi applyOneItem() (trừ theo SL mới) trong 1 transaction → tổng consume luôn = SL gốc.
  • Validation song song FE+BE: mọi guard (≥1 ảnh, tổng giao > 0, không vượt tồn) đều có ở cả 2 phía; FE để UX, BE để an toàn.
  • Số học đơn vị: tồn kho lưu ở base unit; quy đổi sang đơn vị hiển thị qua conversion_factor. Không hardcode đơn vị.
  • Trạng thái rõ ràng: draft → confirmed → delivered/received. Mỗi bước có người/endpoint riêng; không nhảy cóc.
  • Đọc qua eager-load, không tra cứu rời: tên đối tác / tồn kho được BE nhồi sẵn vào response (with() + accessor append), tránh FE tra cứu từ list paginated (fail im lặng).

5. Logic xác nhận giao + tách đơn cốt lõi

Request

POST /v1/orders/{id}/fulfill
{
  "confirmed_by_user_id": 2,         // = user kho đang đăng nhập (cho meta.fulfillment)
  "confirmed_by_name": "Trần Văn B",
  "note": null,
  "items": [                         // SL giao THỰC TẾ từng món
    { "id": 501, "qty_delivered": 5 },
    { "id": 502, "qty_delivered": 0 }   // 0 = bỏ giao món này
  ]
}

Quy tắc (BE enforce — FE cũng mirror)

  • Đơn phải status = confirmed; nếu không → 422 NOT_CONFIRMED.
  • Phải có ≥ 1 ảnh (kind=photo) đã upload trước → nếu không 422 PHOTO_REQUIRED.
  • Mỗi qty_delivered ≥ 0 (được phép = 0). Tổng giao > 0 → nếu tổng = 0: 422 TOTAL_DELIVERED_ZERO.
  • Cho phép giao DƯ (qty_delivered > qty_unit) — không chặn ở BE.
  • Với mỗi món có qty_delivered ≠ qty_unit:
    • Nếu delivered < ordered → gom phần thiếu (ordered − delivered) vào danh sách "còn lại".
    • revertOneItem() (hoàn tồn cũ) → nếu delivered > 0 thì applyOneItem(delivered) (trừ tồn mới); nếu = 0 thì bỏ món khỏi đơn đã giao.
    • Nếu type=purchase → ghi lại inventory batch cho món mới.
  • Nếu có "phần còn lại" → tạo 1 đơn DRAFT mới: cùng type/party, parent_order_id = đơn gốc, notes = "Phần còn lại của đơn hàng {code}".
  • Set đơn gốc: status = delivered|received, completed_at = now, meta.fulfillment = {...}.
  • Tất cả trong 1 DB transaction; lỗi runtime → rollback + 422 FULFILL_FAILED.

Chặn vượt tồn (chỉ FE, chỉ đơn XUẤT kho)

consumesStock = !isPurchase (tức "Xuất kho"). Nếu qty_delivered > stock_unit ở 1 món của đơn xuất → checkbox món đó bị khoá → không tích được → không xác nhận được. Giảm SL ≤ tồn thì mở lại. Đơn nhập chỉ hiển thị tồn, không chặn.

Response

{ "data": {
    "order": Order,              // đơn đã giao (status mới, items đã cập nhật SL)
    "remainder_order": Order|null  // đơn nháp phần còn lại (nếu có)
} }

Logic chi tiết cần port

  • SL giao sửa được: stepper [−] [input] [+], khởi tạo = qty_unit; cho phép vượt SL đặt; không âm.
  • Bộ đếm "giao/đặt" trên card thông tin cập nhật ngay khi chỉnh; gộp theo đơn vị, mỗi đơn vị 1 màu.
  • "Tồn {N}:" trước tên đơn vị mỗi món (từ stock_unit); đỏ khi vượt tồn (đơn xuất).
  • Gate xác nhận = (≥1 ảnh) ∧ (mọi món được tích ∧ không vượt tồn) ∧ (tổng giao > 0).
  • Tích kiểm: mỗi món 1 checkbox; "Tất cả" tích/bỏ hết; checkbox bị khoá nếu món vượt tồn.
  • Ảnh: chọn từ thư viện (ưu tiên) hoặc chụp; upload từng ảnh; lưới 3 cột; xoá được.
  • Vuốt điều hướng: phải→trái = đơn kế (hết → toast); trái→phải = đơn trước (đầu DS → về list). Hiệu ứng trượt. Thứ tự theo danh sách của tab. (Áp dụng 3 tab bán/nhập/hoàn thành, không áp Ảnh chụp.)
  • Xác nhận → gọi fulfill → toast (kèm mã đơn còn lại nếu có) → quay về list.

// VapiApiService.kt
interface VapiApiService {
    @POST("v1/auth/login")
    suspend fun login(@Body body: LoginRequest): VapiResponse<LoginData>

    @GET("v1/orders")
    suspend fun listOrders(
        @Query("status") status: String,
        @Query("per_page") perPage: Int = 100,
    ): VapiResponse<List<OrderDto>>

    @GET("v1/orders/{id}")
    suspend fun getOrder(@Path("id") id: Long): VapiResponse<OrderDto>

    @POST("v1/orders/{id}/fulfill")
    suspend fun fulfill(
        @Path("id") id: Long,
        @Body body: FulfillRequest,
    ): VapiResponse<FulfillResult>

    @GET("v1/attachments")
    suspend fun listAttachments(
        @Query("attachable_type") type: String = "order",
        @Query("attachable_id") id: Long? = null,
        @Query("kind") kind: String = "photo",
        @Query("per_page") perPage: Int = 200,
    ): VapiResponse<List<AttachmentDto>>

    @Multipart
    @POST("v1/attachments")
    suspend fun uploadPhoto(
        @Part("attachable_type") type: RequestBody,
        @Part("attachable_id") id: RequestBody,
        @Part("kind") kind: RequestBody,
        @Part file: MultipartBody.Part,
    ): VapiResponse<AttachmentDto>

    @DELETE("v1/attachments/{id}")
    suspend fun deleteAttachment(@Path("id") id: Long): VapiResponse<Unit>
}

// VapiResponse.kt — wrapper giống ApiResponse của 9chat
data class VapiResponse<T>(val success: Boolean, val data: T?, val message: String? = null, val code: String? = null)
// VapiClient.kt — interceptor gắn X-API-Key + Bearer
class VapiAuthInterceptor(private val tokenStore: VapiTokenStore) : Interceptor {
    override fun intercept(chain: Interceptor.Chain): Response {
        val req = chain.request().newBuilder()
            .header("Accept", "application/json")
            .header("X-API-Key", BuildConfig.VAPI_API_KEY)
        tokenStore.token?.let { req.header("Authorization", "Bearer $it") }
        return chain.proceed(req.build())
    }
}
// Retrofit: baseUrl("https://vapi.vn/api/") + GsonConverterFactory
// Lỗi 4xx: Retrofit ném HttpException → bắt riêng, parse body JSON lấy `code`/`message`
//          (xem memory: Android Retrofit 4xx).
Gotcha Android Retrofit tự ném HttpException cho 4xx/5xx. Để hiện message lỗi từ BE (vd TOTAL_DELIVERED_ZERO, PHOTO_REQUIRED), phải try/catch HttpException rồi parse e.response()?.errorBody() thành JSON.

8.2 DetailViewModel — SL giao, tồn, gate

@HiltViewModel
class WarehouseOrderDetailViewModel @Inject constructor(
    private val repo: WarehouseRepository, private val session: VapiSession,
) : ViewModel() {
    var order by mutableStateOf<OrderDto?>(null); private set
    val delivered = mutableStateMapOf<Long, Double>()   // qty giao theo item
    val checked   = mutableStateMapOf<Long, Boolean>()
    var photos by mutableStateOf<List<AttachmentDto>>(emptyList()); private set

    val isPurchase get() = order?.let { it.type=="purchase"||it.type=="supplier_return" } ?: false
    val consumesStock get() = !isPurchase

    fun isBlocked(it: OrderItemDto): Boolean {
        val s = it.stockUnit ?: return false
        return consumesStock && (delivered[it.id] ?: 0.0) > s + 1e-9
    }
    val hasOverStock get() = order?.items?.any(::isBlocked) ?: false
    val totalDelivered get() = order?.items?.sumOf { (delivered[it.id] ?: 0.0).coerceAtLeast(0.0) } ?: 0.0
    val allChecked get() = order?.items?.takeIf{it.isNotEmpty()}
        ?.all { (checked[it.id]==true) && !isBlocked(it) } ?: false
    val canConfirm get() = photos.isNotEmpty() && allChecked && totalDelivered > 0 && !hasOverStock

    fun load(id: Long) = viewModelScope.launch {
        order = null
        val o = repo.getOrder(id); order = o
        delivered.clear(); checked.clear()
        o.items.forEach { delivered[it.id] = it.qtyUnit; checked[it.id] = false }
        photos = repo.photos(id)
    }
    fun step(it: OrderItemDto, d: Double) {
        delivered[it.id] = ((delivered[it.id] ?: it.qtyUnit) + d).coerceAtLeast(0.0)
    }
    fun confirm(onDone: (remainderCode: String?) -> Unit) = viewModelScope.launch {
        val items = order!!.items.map { DeliveredItem(it.id, (delivered[it.id]?:0.0).coerceAtLeast(0.0)) }
        val res = repo.fulfill(order!!.id, FulfillRequest(session.userId, session.userName, items))
        onDone(res.remainderOrder?.code)
    }
}
