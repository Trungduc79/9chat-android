3. Hệ thống module: Descriptor + Registry

Registry chỉ giữ metadata nhẹ (không khởi tạo gì). Module chỉ "sống" khi user vào.

// Mô tả 1 module quản trị — DỮ LIỆU THUẦN, không giữ state/đối tượng nặng.
data class AdminModule(
    val id: String,                    // "warehouse" | "accounting" | "admin"
    val title: String,                 // "Vận hành kho"
    @DrawableRes val icon: Int,
    val requiredPermissions: List<String>,  // OR: hasAny(...) → đủ 1 là vào được
    val route: String,                 // route gốc của nested graph module
    val registerGraph: NavGraphBuilder.(NavHostController) -> Unit, // đăng ký màn (lazy)
)

// Đăng ký tĩnh — chỉ là danh sách metadata, KHÔNG tạo ViewModel/đối tượng.
object ModuleRegistry {
    val all: List<AdminModule> = listOf(
        WarehouseModule.descriptor,
        AccountingModule.descriptor,
        AdminModule_.descriptor,
    )
    fun visibleFor(perm: PermissionStore): List<AdminModule> =
        all.filter { perm.isBypass || perm.hasAny(*it.requiredPermissions.toTypedArray()) }
}

// Mỗi module tự khai báo descriptor + cách đăng ký màn của nó.
object WarehouseModule {
    val descriptor = AdminModule(
        id = "warehouse",
        title = "Vận hành kho",
        icon = R.drawable.ic_module_warehouse,
        requiredPermissions = listOf("order.view_all", "order.update_status"),
        route = "module/warehouse",
        registerGraph = { nav -> warehouseGraph(nav) },  // xem guide kho
    )
}
Thêm module mới = thêm 1 package + 1 AdminModule vào ModuleRegistry.all. Không đụng tab Khám phá, không đụng module khác → đúng tinh thần "module riêng biệt".

4. Phân quyền theo permission code

1 app-module gom nhiều permission code (RBAC 9chat có sẵn). hasAny = đủ 1 quyền là vào được.

App-modulerequiredPermissions (hasAny)Role thường có
Vận hành khoorder.view_all, order.update_status, supplier.view_purchase_orderstaff, shipper, manager
Kế toándebt.view_all, invoice.view_all, report.financialaccountant, manager
Quản trị tổngsystem.permission_manage, system.settings (hoặc bypass_all)owner, manager
Cần backend 9chat: nếu muốn module "Kho" tách bạch khỏi quyền order CSKH, nên thêm nhóm permission warehouse.* (vd warehouse.fulfill) vào migration RBAC + gán cho role kho. Trước mắt tái dùng order.*. (Quyết định ở §10.)

Default-DENY sẵn có: chưa fetch quyền hoặc lỗi mạng → has()=false → không hiện module (an toàn). Tài khoản customer (mặc định) không có quyền nào trong list → tab Khám phá trống.

5. ⭐ Vòng đời & chống thất thoát tài nguyên TRỌNG TÂM

Nguyên tắc: module chỉ tiêu tốn tài nguyên KHI đang mở; rời module = dọn sạch. Cụ thể:

5.1 Lazy — không khởi tạo cho tới khi vào

  • Registry chỉ là data class (id/title/icon/perms/route) → gần như 0 chi phí.
  • ViewModel + Composable của module chỉ được tạo khi navigate(module.route). Compose Navigation lo việc này.
  • VapiClient (Retrofit/OkHttp cho vapi) khởi tạo lazy lần đầu có module dùng; OkHttp dùng connection pool nhàn rỗi tự thu hồi.

5.2 Scoped ViewModel → tự huỷ khi rời module

// ViewModel scope vào BackStackEntry của nested graph module.
// Khi popBackStack khỏi graph → onCleared() → viewModelScope.cancel()
// → mọi coroutine (poll, collect, upload) bị huỷ ngay. KHÔNG để VM ở app-level.
@Composable fun moduleVm(nav: NavHostController, graphRoute: String): XxxViewModel {
    val entry = remember(it) { nav.getBackStackEntry(graphRoute) }
    return hiltViewModel(entry)
}

5.3 DisposableEffect cho mọi listener/sensor

// Bất kỳ thứ gì "đăng ký" (socket event, location, camera, broadcast receiver)
// PHẢI gỡ trong onDispose. Ví dụ nếu module cần realtime:
DisposableEffect(Unit) {
    val l: (Array<Any>) -> Unit = { /* ... */ }
    container.socket.on("warehouse_update", l)
    onDispose { container.socket.off("warehouse_update", l) }   // ⬅ bắt buộc
}

5.4 Quy tắc cứng (review checklist mỗi module)

Tài nguyênQuy tắc
CoroutineChỉ dùng viewModelScope (scoped) — KHÔNG GlobalScope. Tự huỷ khi rời.
State / cacheGiữ trong ViewModel của module; KHÔNG nhồi vào AppContainer (sống suốt đời app).
Polling / timerCấm polling nền. Cần làm mới → refetch khi mở / pull-to-refresh / sự kiện. Nếu buộc poll → chạy trong repeatOnLifecycle(STARTED), dừng khi background.
Ảnh (Coil)Dùng Coil (cache có giới hạn sẵn). KHÔNG giữ Bitmap trong VM. Nén ảnh trước upload rồi giải phóng.
Background serviceMặc định KHÔNG. Nếu module thật sự cần (vd quét) → start khi vào, stopSelf() khi ra.
Socket realtimeDùng chung container.socket của 9chat; module chỉ on/off trong DisposableEffect. KHÔNG mở socket riêng.
Observer/FlowcollectAsStateWithLifecycle() (dừng thu khi màn không STARTED).

5.5 "Không load module khác" nghĩa là gì ở đây

Với feature-package: code các module compile chung trong APK, nhưng runtime KHÔNG khởi tạo/không chạy module không được cấp quyền (không tạo VM, không gọi API, không observer). → đạt mục tiêu "không ngốn tài nguyên". Nếu sau này muốn không cả tải code module (giảm dung lượng tải) → nâng lên Gradle dynamic-feature (Play Feature Delivery) — để dành Phase sau.

6. Tab Khám phá (hub)

// HomeScreen.kt — thay PlaceholderTab("Khám phá") ở case selectedTab == 2:
2 -> ExploreScreen(onOpenModule = { route -> navigateTo(AppDestination.Module(route)) })

@Composable fun ExploreScreen(onOpenModule: (String) -> Unit) {
    val container = (LocalContext.current.applicationContext as App).container
    val perms by container.permissions.state.collectAsStateWithLifecycle()
    LaunchedEffect(Unit) { container.permissions.refresh() }   // đảm bảo có quyền mới

    val modules = remember(perms) { ModuleRegistry.visibleFor(container.permissions) }
    if (modules.isEmpty()) { EmptyState("Chưa có chức năng quản trị"); return }

    LazyVerticalGrid(columns = GridCells.Fixed(2)) {
        items(modules, key = { it.id }) { m ->
            ModuleCard(m) { onOpenModule(m.route) }   // chỉ render card; module CHƯA khởi tạo
        }
    }
}

Thêm AppDestination.Module(val route:String) vào sealed class + nhánh when trong navigateTo để host nested graph tương ứng (registry tìm module theo route → registerGraph).

7. Cầu nối dữ liệu: vapi gateway

Mô hình: app → vapi trực tiếp. vapi đóng vai API gateway, định danh client qua API key. Tạo 1 api_key mới trong vapi: tên "9chat-android", cấp scope cho các module (orders, attachments, debt, invoice…). App nhúng key qua BuildConfig.VAPI_API_KEY (không hardcode trong source).

  • Định danh app: header X-API-Key: <9chat-android key> → vapi audit created_by_api_key_id = app này.
  • Định danh user: truyền danh tính user 9chat trong payload nơi cần (vd confirmed_by_user_id, confirmed_by_name ở fulfill) — vapi lưu để truy vết NV.
  • Phân quyền: ai được vào module = RBAC 9chat (UI gate). vapi tin app qua API key (giai đoạn này); siết per-user ở vapi là việc sau.

Lớp mạng vapi + DTO + repository: xem chi tiết file warehouse-ops-android-guide.html §7. Mọi module quản trị dùng chung 1 VapiClient (lazy singleton trong container hoặc Hilt).

9. Lộ trình triển khai (phased)

Bước 0 — chốt backend: tạo api_key "9chat-android" trong vapi (+scope); quyết định dùng order.* hay thêm warehouse.* trong RBAC 9chat (§10).
Bước 1 — khung module (không phụ thuộc vapi): AdminModule + ModuleRegistry + ExploreScreen hub + AppDestination.Module. Tạo 2-3 descriptor với màn rỗng. → Kiểm chứng gate theo quyền chạy đúng (đổi role → đổi module hiển thị).
Bước 2 — lớp vapi dùng chung: VapiClient + interceptor API key + DTO + repository (lazy).
Bước 3 — Module Kho: port theo guide HTML (login bỏ qua nếu dùng API key; orders 4 tab; detail + swipe + fulfill). Áp checklist tài nguyên §5.4.
Bước 4 — Module Kế toán / Admin tổng: lặp lại pattern.
Bước 5 — siết: per-user enforcement ở vapi; (tuỳ chọn) nâng lên dynamic-feature nếu cần giảm size tải.

10. Việc cần backend / cần chốt thêm

  • vapi: tạo api_key "9chat-android" + cấp scope (orders, attachments, debt, invoice, report…). Đây là điều kiện để Bước 2+ chạy.
  • RBAC 9chat: dùng tạm order.* cho Kho, hay thêm nhóm warehouse.*? (Khuyến nghị: thêm warehouse.fulfill + gán role kho để tách bạch CSKH vs kho.)
  • Định danh user qua vapi: hiện truyền by_user_id/by_name trong payload. Nếu cần map cứng user 9chat ↔ user vapi (cho công nợ/đối soát theo NV) → cần bảng map ở vapi (việc sau).
  • Icon module: cần 3 drawable ic_module_warehouse/accounting/admin.

Đây là phương án CHỐT TRƯỚC KHI CODE. Sau khi anh duyệt §10, em bắt đầu Bước 1 (khung module + gate) — phần thuần Android, chưa cần backend, để anh thấy cơ chế phân quyền chạy thật.
